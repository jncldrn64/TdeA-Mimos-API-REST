## 2. HITOS ALCANZADOS EN ESTA SESIÓN

### Entidad `Envio` — completada y validada
Séptima entidad del sistema. Depende de `Venta` vía FK obligatoria (`venta_id`).

Se generaron e integraron los 6 archivos de la arquitectura:

| Archivo | Ruta | Responsabilidad |
|---|---|---|
| `Envio.java` | `entity/` | Mapea la tabla `Envios` con FK hacia `Ventas` |
| `EnvioRepository.java` | `repository/` | CRUD + `findByVenta_Id()` + `findByEstadoEnvio()` |
| `EnvioDTO.java` | `dto/` | Respuesta pública (ventaId como Long, no objeto completo) |
| `CrearEnvioDTO.java` | `dto/` | Entrada del POST (campos que envía el cliente) |
| `EnvioService.java` | `service/` | Lógica de negocio + validaciones de FK y unicidad |
| `EnvioController.java` | `controller/` | 5 endpoints HTTP REST |

### Decisiones técnicas tomadas y por qué

**`insertable = false` (sin `updatable = false`) en `estadoEnvio`**
A diferencia de `fecha` en `Venta` o `createdAt` en `Usuario` que nunca se modifican, el estado del envío sí cambia durante el ciclo de vida (`'preparando'` → `'despachado'` → `'entregado'`). Por eso solo se bloquea la inserción (la DB asigna `'preparando'` por DEFAULT) pero se permite la actualización posterior.

**Fechas sin `insertable = false`**
`fechaDespacho`, `fechaEntregaEstimada` y `fechaEntregaReal` no tienen DEFAULT en SQL Server — nacen como NULL por diseño y se rellenan conforme avanza el proceso logístico. Hibernate sí las incluye en el INSERT (con valor null), lo cual es el comportamiento correcto.

**Validación de unicidad venta→envío en el Service**
Antes de insertar, el Service verifica que la venta no tenga ya un envío asociado. Si lo tiene, devuelve 400 con mensaje claro en lugar de dejar que la DB explote con un error de FK o lógica inconsistente.

**`findByVenta_Id()` con doble guion bajo**
Spring genera el JOIN automáticamente a partir del nombre del método. El `_Id` final navega la relación `@ManyToOne` hasta la PK de `Venta`. El SQL generado usa `LEFT JOIN dbo.Ventas ON ventas.id = envios.venta_id`, exactamente como se verificó en consola.

### Endpoints disponibles y validados

| Método | URL | Descripción | Código | Estado |
|---|---|---|---|---|
| `GET` | `/api/envios` | Lista todos los envíos | 200 | ✅ Validado |
| `GET` | `/api/envios/{id}` | Busca envío por ID | 200 / 404 | ✅ Validado |
| `GET` | `/api/envios/venta/{ventaId}` | Envío de una venta específica | 200 / 404 | ✅ Validado |

## 3. 🚨 PROTOCOLO ESTRICTO DE PRUEBAS EN POWERSHELL 🚨

**OBLIGATORIO para la IA de relevo:** El entorno usa PowerShell 5.x en Windows. Cualquier intento de pasar JSON inline con `-d '{"key":"value"}'` o variables `$body` fallará con error 400.

**Único método validado:**
```powershell
[System.IO.File]::WriteAllText("$env:TEMP\payload.json", '{"clave":"valor"}')
curl.exe -X POST http://localhost:8081/api/endpoint -H "Content-Type: application/json" -d "@$env:TEMP\payload.json"
```

1. Instalar SQL Server. Ejecutar el script base para crear `MimosDemo`.
2. Crear usuario de acceso:
   ```sql
   CREATE LOGIN api_mimos_user WITH PASSWORD = 'MimosDev2026*';
   USE [MimosDemo];
   CREATE USER api_mimos_user FOR LOGIN api_mimos_user;
   ALTER ROLE db_owner ADD MEMBER api_mimos_user;
   ```

3. Ajustar `spring.datasource.url` en `application.properties`. Sin Podman/VM usar `127.0.0.1`.
4. Desde `/api_rest/` en consola: `.\mvnw clean spring-boot:run`

## 7. NOTA PARA LA IA QUE TOME EL RELEVO — ENTIDAD `PQRS`

La única entidad pendiente es **`PQRS`** (Peticiones, Quejas, Reclamos y Sugerencias).

**Dependencias:**
- FK obligatoria hacia `Usuarios` (`usuario_id`)
- FK opcional hacia `Ventas` (`venta_id` puede ser NULL — una PQRS puede no estar asociada a una venta)

**Campos con DEFAULT en DB (usar `insertable = false`):**
- `estado` → DEFAULT `'abierto'`
- `fecha_creacion` → DEFAULT `GETDATE()`

**Campos que nacen NULL y se rellenan después (NO usar `insertable = false`):**
- `respuesta` — la escribe el admin al gestionar la PQRS
- `fecha_respuesta` — se registra cuando el admin responde

**Regla de negocio importante:** Si se envía un `ventaId` en el POST, debe existir en la DB. Si es null, se acepta sin validación.

**Protocolo de pruebas:** Usar siempre `WriteAllText` + `curl.exe`. Ver sección 3.
