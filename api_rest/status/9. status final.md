**Documento de Arquitectura, Validación, Resolución de Errores y Estándares HTTP**

## 1. EL PROPÓSITO GLOBAL: ¿POR QUÉ ESTAMOS HACIENDO ESTO?
Para entender todo, hay que mirar el panorama completo. Mimos tenía (o tiene, en producción) una aplicación monolítica en Node.js, donde el frontend, el backend y la lógica estaban fuertemente acoplados. 

El objetivo de este proyecto fue crear una **API REST moderna en Java 17 con Spring Boot**, conectada a una base de datos SQL Server normalizada en Tercera Forma Normal (3FN) llamada `MimosDemo`. 

**¿Por qué aislar esto?** Porque una API (Application Programming Interface) actúa como un "mesero" en un restaurante. El cliente (Frontend / Postman / curl) no entra a la cocina (Base de Datos) a prepararse la comida. El cliente le entrega una "orden" (Petición HTTP) al mesero (API), el mesero va a la cocina, valida que haya ingredientes, prepara el plato de forma segura y se lo entrega al cliente (Respuesta JSON). Esto permite que el día de mañana Mimos pueda tener una página web, una app en Android y una app en iOS, y las tres consumirán exactamente la misma API sin reescribir la lógica de negocio.

## 2. ¿POR QUÉ ES CRÍTICO VALIDAR EL BACKEND ANTES DE TOCAR EL FRONTEND?
Validar la API exhaustivamente antes de conectar un frontend en JavaScript (React, Angular, Vue, o Vanilla JS) es una regla de oro inquebrantable. 

**Si no validas el backend primero y conectas el frontend de golpe, te vas a enfrentar al "Infierno del Debugging".** Cuando algo falle, no sabrás si el error está en:
1. Tu código de JavaScript (Frontend).
2. La red (CORS, URLs mal escritas).
3. El Controlador de Spring Boot.
4. La lógica del Servicio (Service).
5. El Repositorio (Hibernate / JPA).
6. La Base de Datos (Restricciones, Llaves foráneas).

Al validar cada endpoint manualmente con Postman o `curl`, estamos aislando las capas. Si `curl` recibe un código `201 Created`, sabemos con 100% de certeza que el Backend y la Base de Datos son perfectos. Si luego el Frontend falla, sabemos que la culpa es exclusiva del Frontend.

## 3. EL DISEÑO DE LA BASE DE DATOS Y LAS ENTIDADES INTERMEDIAS
Te preguntarás: *¿Por qué tenemos tablas "entre medias" como `CarritoItem` o `DetalleVenta` en lugar de guardar todo en una sola tabla?*

Esto se debe a las **Relaciones de Muchos a Muchos (N:M)** y la necesidad imperiosa de conservar el **historial financiero**.
* **CarritoItem:** Un `Carrito` puede tener muchos `Productos`, y un `Producto` puede estar en los carritos de muchas personas. No puedes poner una columna "id_producto" en la tabla Carrito porque solo cabría un producto. La tabla intermedia `CarritoItem` resuelve esto enlazando 1 Carrito con 1 Producto, además de guardar la `cantidad` y el `precio_snapshot`.
* **DetalleVenta:** Similar al carrito. ¿Por qué guardamos el precio en el detalle de la venta si el producto ya tiene precio? Porque los precios cambian. Si hoy compras un helado a $5,000 y mañana sube a $7,000, tu factura histórica no puede mágicamente decir que pagaste $7,000. El `precio_unitario` en el `DetalleVenta` congela la realidad en el tiempo.

## 4. ANATOMÍA DE UNA PETICIÓN HTTP: HEADERS, BODY, Y PATH
Para comunicarte con la API, envías "Paquetes" (Peticiones HTTP). Imagina que envías una carta por correo físico:

1. **La URL y el Path (El destinatario):** Es la dirección exacta. `http://localhost:8081/api/usuarios/1`. El `1` al final es un "Path Variable" (Variable de Ruta). Le dice a la API exactamente sobre qué recurso estás operando.
2. **El Header o Cabecera (El sobre):** Son metadatos invisibles para el usuario final, pero cruciales para el servidor. El header más crítico que usamos es `Content-Type: application/json`. Esto le grita a Spring Boot: *"¡Oye! Lo que viene adentro del sobre está escrito en idioma JSON, no en texto plano ni en XML"*.
3. **El Body o Cuerpo (La carta adentro del sobre):** Es la carga útil (Payload) de tu petición. Contiene el JSON con los datos complejos que quieres guardar. Ejemplo: `{"nombre": "Helado", "precio": 5000}`. 

**Diferencia clave:** Lees de la ruta (`@PathVariable`) cuando solo necesitas un identificador simple (como un ID para buscar o borrar). Lees del cuerpo (`@RequestBody`) cuando necesitas recibir un objeto complejo con múltiples propiedades (como crear un Producto entero).

## 5. LOS MÉTODOS HTTP (GET, POST, PUT, DELETE) Y NUESTRO ENFOQUE
En una API REST pura, los métodos HTTP se mapean a las operaciones CRUD (Create, Read, Update, Delete).

* **GET (Leer):** Es el método más seguro. Nunca modifica nada en la base de datos. Sirve para consultar. Se envía sin Body. Lo usamos exhaustivamente (`/api/productos`, `/api/ventas`).
* **POST (Crear):** Se usa para insertar un nuevo registro. Requiere un Body con el JSON. Lo usamos para registrar usuarios, productos, ventas y quejas.
* **DELETE (Eliminar):** Destruye un registro usando su ID en la ruta. 
* **PUT / PATCH (Actualizar):** *¿Por qué NO lo usamos en Mimos casi en absoluto?* Porque en un sistema transaccional comercial, actualizar datos históricos es un riesgo de auditoría masivo. No haces un "PUT" a una Venta para cambiar su total. Si la venta estuvo mal, la anulas y haces otra. Al omitir el PUT, blindamos la aplicación contra modificaciones maliciosas de registros financieros.

## 6. EL DICCIONARIO DE ANOTACIONES DE SPRING BOOT
Para que Java entienda todo este tráfico web, usamos anotaciones (las palabras que empiezan con `@`):

* **`@RestController`:** Convierte una clase normal en un semáforo web. Le dice a Spring que todo lo que devuelva esta clase debe convertirse automáticamente a JSON.
* **`@RequestMapping("/api/...")`:** Define la ruta base para todo el controlador.
* **`@GetMapping`, `@PostMapping`, `@DeleteMapping`:** Mapean un verbo HTTP específico a un método Java específico. *(Recordemos el fallo que tuvimos cuando escribimos `@getMapping` en minúscula: ¡Java es case-sensitive y la compilación explotó!)*.
* **`@RequestBody`:** Atrapa el JSON del Body (la carta) y lo convierte en un objeto Java (`DTO` o `Entity`).
* **`@PathVariable`:** Extrae un pedazo de la URL (el destinatario) y lo mete en una variable Java (ej. `Long id`).

## 7. CURL VS POSTMAN: LA MASACRE DE POWERSHELL
Para enviar estas peticiones manualmente, tienes dos opciones: herramientas gráficas como **Postman**, o la línea de comandos con **`curl`**. Y aquí es donde vivimos nuestra mayor pesadilla.

**El contexto del desastre:** El usuario estaba en un entorno Windows usando PowerShell nativo. PowerShell tiene una forma muy hostil de interpretar las comillas simples (`'`) y dobles (`"`) y las barras invertidas (`\`).
Cuando intentábamos enviar un JSON complejo directamente en la terminal usando variables como `-d $body` o `-d '{"key":"value"}'`, PowerShell destruía el JSON antes de enviarlo. El servidor Spring Boot recibía un texto corrupto y nos escupía errores `400 Bad Request` con el infame mensaje: `JSON parse error: Unexpected character '' (code 92)`.

**La Solución Definitiva (El Script Inteligente):**
Para vencer a PowerShell, creamos un estándar inquebrantable en el proyecto. Prohibimos enviar JSONs directamente en el comando. En su lugar, usamos `[System.IO.File]::WriteAllText` para obligar a Windows a escribir el JSON en un archivo temporal de texto plano UTF-8 en el disco duro, y luego le dijimos a `curl.exe` que leyera ese archivo usando el símbolo `@` (`-d "@$env:TEMPrchivo.json"`). ¡Esto salvó el proyecto!

## 8. LA GRAN BITÁCORA DE ERRORES Y TROUBLESHOOTING
No nos cortaremos ni un pelo. Este proyecto fue una batalla arquitectónica y superamos obstáculos enormes. Aquí está el muro de la fama de nuestros errores, para que cualquier IA o humano que lea esto en el futuro sepa qué pasó:

1. **La colisión del Puerto 8080:** El sistema legacy de Node.js ya estaba ocupando el puerto 8080. Tuvimos que mover la API de Spring Boot al puerto `8081` en el `application.properties`.
2. **El laberinto de Podman y el Localhost:** La base de datos SQL Server estaba en un contenedor Podman en un host Linux (IP `10.10.1.1`), mientras que el desarrollo se hacía en una VM Windows. Tuvimos errores `Connection refused` porque la VM buscaba la DB dentro de sí misma (`127.0.0.1`). Lo solucionamos apuntando el JDBC explícitamente a la IP del contenedor.
3. **Hibernate y los nombres en minúscula:** Hibernate intentaba buscar tablas en minúscula en esquemas inexistentes (`Schema-validation: missing table`). Tuvimos que inyectar la estrategia `physical-strategy` y definir `default_schema=dbo` en las properties.
4. **El Falso `curl` de PowerShell:** Descubrimos que la palabra `curl` en PowerShell es un alias del comando `Invoke-WebRequest`, el cual funciona totalmente distinto. Tuvimos que forzar el uso de `curl.exe` nativo en todos los scripts.
5. **El Error de Sintaxis `@getMapping`:** Un despiste humano (escribir la anotación en minúscula) hizo que Maven arrojara 7 errores `cannot find symbol`. Nos enseñó la importancia del Case Sensitivity en Java.
6. **El Bug de la Categoría Nula (La Discrepancia del DTO):** La IA de relevo diseñó `CrearProductoDTO` esperando un campo plano `"idCategoria": 1`, pero nuestros scripts enviaban un objeto anidado `"categoria": {"idCategoria": 1}`. Como resultado, Spring Boot guardaba los productos con la llave foránea en `null`. Lo solucionamos ajustando el JSON del script de pruebas para igualar el "Flat DTO".
7. **El Falso Error 400 en Envíos:** Cuando intentamos simular un Envío para la Venta 1, nos dio `400 Bad Request`. Pensamos que era un bug, pero resultó ser la validación de negocio de la IA de relevo: ¡La Venta 1 ya tenía un envío despachado y la lógica impidió crear un duplicado!
