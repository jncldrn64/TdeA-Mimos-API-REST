## 2. HITOS ALCANZADOS EN ESTA SESIÓN

### Entidad `Producto` — completada y validada (GET)
Esta es la cuarta entidad del sistema y la primera con múltiples endpoints especializados. Depende de `Categoria` vía FK opcional (`id_categoria` puede ser NULL).

Se generaron e integraron los 6 archivos de la arquitectura:

| Archivo | Ruta | Responsabilidad |
|---|---|---|
| `Producto.java` | `entity/` | Mapea la tabla `Productos` con FK opcional a `Categorias` |
| `ProductoRepository.java` | `repository/` | CRUD + `findByEstaActivoTrue()` + `findByCategoria_IdCategoria()` |
| `ProductoDTO.java` | `dto/` | Respuesta pública (categoría como nombre, no como objeto) |
| `CrearProductoDTO.java` | `dto/` | Entrada del POST (campos que envía el cliente) |
| `ProductoService.java` | `service/` | Lógica de negocio, mapeo Entity→DTO, validación de FK |
| `ProductoController.java` | `controller/` | 5 endpoints HTTP REST |

### Decisiones técnicas tomadas y por qué

**FK opcional (`nullable = true`) en la relación con `Categoria`**
A diferencia de `Usuario→Rol` (donde el rol es obligatorio), la tabla `Productos` permite `id_categoria NULL`. Esto refleja fielmente el esquema SQL. Un producto puede existir sin categoría asignada, y el Service lo maneja devolviendo `null` en el campo `categoria` del DTO en ese caso.

**Dos métodos de consulta derivados en el Repository**
- `findByEstaActivoTrue()`: genera `WHERE esta_activo = 1`. Sirve para el catálogo público donde no deben aparecer productos dados de baja.
- `findByCategoria_IdCategoria(Long id)`: genera un `LEFT JOIN` automático con `Categorias` filtrando por `id_categoria`. El doble guion bajo (`_`) es la sintaxis de Spring para navegar relaciones en nombres de métodos.

**`BigDecimal` para `precioUnitario`**
Los tipos `double` y `float` tienen errores de precisión binaria en operaciones decimales. Por ejemplo, `8500.10 + 0.20` puede dar `8500.299999...` con double. `BigDecimal` garantiza precisión exacta para dinero.

**`insertable = false, updatable = false` en `fechaIngreso` y `estaActivo`**
Ambos campos tienen `DEFAULT` en SQL Server (`GETDATE()` y `1` respectivamente). Marcándolos así, Hibernate los omite en el INSERT y deja que la DB los asigne. El `null` que aparece en la respuesta del POST es esperado — mismo comportamiento documentado en `Usuario` con `estado` y `createdAt`.

**Dos DTOs (`ProductoDTO` + `CrearProductoDTO`)**
Mismo patrón que `Usuario`: el DTO de entrada no incluye campos autogenerados por la DB (`fechaIngreso`, `estaActivo`, `idProducto`). El cliente solo envía lo que debe enviar.

## 3. BITÁCORA DE ERRORES Y SOLUCIONES (ACUMULADA)

| Error / Comportamiento | Causa Diagnosticada | Solución Implementada |
|---|---|---|
| **Fallo NetBeans `${start-class}`** | El IDE asume rutas erróneas. | Ejecutar siempre por consola: `.\mvnw spring-boot:run`. |
| **`Connection refused` (127.0.0.1)** | La VM buscaba la DB localmente. | JDBC apunta a `10.10.1.1` (Podman). |
| **`Schema-validation: missing table`** | Hibernate asume nombres en minúscula. | `physical-strategy` estándar en `application.properties`. |
| **`package ...bcrypt does not exist`** | Faltaba dependencia `spring-security-crypto` en `pom.xml`. | Agregar dependencia y correr `.\mvnw dependency:resolve`. |
| **`JSON parse error: Unexpected character '\'`** | PowerShell interpola comillas escapadas en `-d` de curl. | Usar variable `$body` o archivo temporal. |
| **`class CrearProductoDTO should be declared in CrearProductoDTO.java`** | El archivo fue guardado con nombre incorrecto (`GenerarProductoDTO.java`). | Renombrar el archivo para que coincida exactamente con el nombre de la clase. |
| **`cannot find symbol: method getNombre()` en todos los Services** | Lombok dejó de procesar las entidades al recompilar tras el archivo mal nombrado. | Correr `.\mvnw clean spring-boot:run` para limpiar `target/` y recompilar desde cero. |
| **`cannot find symbol: class LocalDateTime` en `CrearProductoDTO`** | Al renombrar el archivo se perdió el `import java.time.LocalDateTime`. | Agregar manualmente el import faltante al inicio del archivo. |
| **`Out-File: utf8NoBOM not valid`** | PowerShell 5.x no soporta el encoding `utf8NoBOM`. | Usar `[System.IO.File]::WriteAllText(path, contenido)` — escribe UTF-8 sin BOM en cualquier versión de PowerShell. |
| **`Couldn't read data from file` en curl con `@archivo`** | El archivo `body.json` no se creó porque el comando anterior falló. | Crear el archivo primero con `WriteAllText`, luego llamar a `curl.exe` con `@ruta`. |

### Patrón definitivo para POST en PowerShell (cualquier versión)
```powershell
[System.IO.File]::WriteAllText("$env:TEMP\body.json", '<JSON aquí>')
curl.exe -X POST <URL> -H "Content-Type: application/json" -d "@$env:TEMP\body.json"
```

## 6. GUÍA DE DESPLIEGUE LOCAL (NUEVA MÁQUINA)

1. Instalar SQL Server. Ejecutar el script base para crear `MimosDemo`.
2. Crear usuario de acceso:
   ```sql
   CREATE LOGIN api_mimos_user WITH PASSWORD = 'MimosDev2026*';
   USE [MimosDemo];
   CREATE USER api_mimos_user FOR LOGIN api_mimos_user;
   ALTER ROLE db_owner ADD MEMBER api_mimos_user;
   ```
3. Ajustar `spring.datasource.url` en `application.properties`. Sin Podman/VM, cambiar `10.10.1.1` por `127.0.0.1`.
4. Desde `/api_rest/` en consola: `.\mvnw spring-boot:run`
