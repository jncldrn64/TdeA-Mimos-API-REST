## 2. HITOS ALCANZADOS EN ESTA SESIÓN

### Entidades `Venta` y `DetalleVenta` — completadas y validadas
Se abordó el sexto y más crítico bloque del e-commerce. La facturación funciona correctamente.

Se generaron e integraron los 10 archivos de la arquitectura correspondientes a las dos entidades (Entity, Repository, Service, Controller, DTO).

**Decisiones técnicas y validaciones:**
- **Relaciones Opcionales:** `procesadoPor` en `Venta` se mapeó permitiendo valores nulos para representar ventas automáticas (online) sin intervención manual de un administrador.
- **Precisión Financiera:** Se utilizó rigurosamente `BigDecimal` para `total` y `precioUnitario`, previniendo errores de redondeo de punto flotante.
- **Pruebas Nativas:** Las pruebas de POST confirmaron el funcionamiento de ambas tablas y sus llaves foráneas (`id` 1 para venta y detalle devueltos exitosamente). 
- **Campos Autogenerados:** En el DTO de respuesta, campos como `fecha` y `estado` retornan `null` intencionalmente al hacer POST (la DB asigna `GETDATE()` y `'pendiente'`), optimizando la operación sin requerir un re-query inmediato por parte de Hibernate.

## 3. 🚨 PROTOCOLO ESTRICTO DE PRUEBAS EN POWERSHELL 🚨

**IMPORTANTE PARA LA IA DE RELEVO:** El usuario opera PowerShell en Windows. PowerShell corrompe sistemáticamente los JSON al pasarlos por consola usando variables (`$body`) o barras invertidas escapadas. **NUNCA** le des comandos con `-d '{"key":"value"}'`.

**LA ÚNICA FORMA APROBADA Y VALIDADA PARA PRUEBAS (POST/PUT):**
```powershell
# 1. Escribir JSON a archivo temporal (OBLIGATORIO)
[System.IO.File]::WriteAllText("$env:TEMP\payload.json", '{"clave": "valor", "anidado": {"id": 1}}')

# 2. Ejecutar curl.exe leyendo el archivo (OBLIGATORIO usar curl.exe)
curl.exe -X POST http://localhost:8081/api/endpoint -H "Content-Type: application/json" -d "@$env:TEMP\payload.json"
```
